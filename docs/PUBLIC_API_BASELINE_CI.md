# Canonical public API baseline and CI enforcement

Phase 0.9.16 converts the deterministic public API snapshot introduced in Phase 0.9.15 into a permanent compatibility gate for the 0.10.0 development series.

## Contract ownership

The committed baseline is derived release evidence. Package source, `NAMESPACE`, and generated documentation remain authoritative for the current interface. The baseline must never be edited by hand to make a check pass.

## CI policy

Every pull request installs the package, generates a deterministic current snapshot, and compares it with `inst/api-contract/public-api-baseline.tsv`.

Blocking changes include:

- removed exports or registered S3 methods;
- removed public arguments;
- reordered or newly inserted required arguments;
- changed public argument defaults.

New exports and optional arguments are advisory findings. They require explicit review and an intentional baseline refresh before release.

CI uploads the current snapshot, findings, summary, and metadata as the `public-api-contract` artifact.

## Refreshing the baseline

After intentionally approving a compatible public API change, install the package from the candidate branch and run:

```bash
Rscript tools/update-public-api-baseline.R
Rscript tools/check-public-api-contract.R
```

The refresh command records the installed package version and entry count. Review the complete baseline diff before committing it. A baseline version mismatch is a CI failure.

## Bootstrap procedure

The Phase 0.9.16 branch initially runs in bootstrap mode only while the first exact installed-package snapshot is generated by CI. That artifact is then committed as the canonical 0.10.0 baseline and bootstrap tolerance is removed before the phase is completed.
