# Reproducible container images

Tagged popgenVCF releases publish OCI images to GitHub Container Registry (GHCR). Each image is built from the exact Git tag, validated during the image build, and tested again after publication by its immutable digest.

## Choose an image identity

Use a shell variable to keep commands consistent:

```bash
export POPGENVCF_IMAGE="ghcr.io/duceppemo/popgenvcf:latest"
```

The floating `latest` tag is suitable for current development and evaluation. For an analysis that must be reproducible, set the variable to a published version or, preferably, the canonical digest recorded in the GitHub Release assets:

```bash
export POPGENVCF_IMAGE="ghcr.io/duceppemo/popgenvcf@sha256:<digest>"
```

A digest identifies the exact OCI manifest and cannot move to another image.

## Pull and inspect the image

```bash
docker pull "$POPGENVCF_IMAGE"
docker image inspect "$POPGENVCF_IMAGE"
```

Record the resolved digest together with the analysis configuration and input checksums.

## Generate a configuration

```bash
docker run --rm \
  --user "$(id -u):$(id -g)" \
  -e HOME=/tmp \
  -v "$PWD:/data" \
  "$POPGENVCF_IMAGE" \
  --write-config /data/analysis.yml
```

## Run an analysis

```bash
docker run --rm \
  --user "$(id -u):$(id -g)" \
  -e HOME=/tmp \
  -v "$PWD:/data" \
  "$POPGENVCF_IMAGE" \
  --config /data/analysis.yml
```

Paths in the configuration must refer to container-visible paths under `/data`. Running as the host user prevents root-owned result files on Linux.

## Release tags

A stable release can publish:

- the full version;
- major/minor and major aliases;
- a full source-commit tag prefixed with `sha-`;
- `latest` for the newest non-prerelease release.

Version aliases are convenient, but only the digest is immutable. Confirm the exact aliases in the corresponding GitHub Release instead of assuming that an unreleased development version is already published.

## Release metadata and attestations

Each published GitHub Release is expected to retain:

- `container-metadata.json`, containing image name, release version, source commit, tags, and canonical digest;
- `container-digest.txt`, a compact digest record for scripts and archives;
- OCI provenance and SBOM attestations generated by the container build;
- release checksums linking the image evidence to the source revision.

Inspect attestations with approved OCI-aware tooling such as Docker Buildx, GitHub attestations, or `cosign`. The presence of an SBOM or provenance attestation establishes artifact traceability; it does not approve a scientific dataset or interpretation.

## Apptainer and Singularity

GHCR images can be consumed on HPC systems:

```bash
apptainer pull popgenvcf.sif \
  "docker://${POPGENVCF_IMAGE}"
```

For archival use, set `POPGENVCF_IMAGE` to the immutable digest and record a checksum for the generated SIF as a separate artifact.

## Verification performed by CI

Before publication, the image build runs package and scientific validation. After the image is pushed, CI pulls the exact digest and verifies:

1. the installed package version matches `DESCRIPTION`;
2. the command-line entry point responds successfully;
3. deterministic scientific validation passes;
4. the published digest and source identity match the release metadata.

Container publication is triggered from release automation. A development branch or version string must not be described as a published container release until the image and its immutable metadata exist.
